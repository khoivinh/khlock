## Flag fixes (this session, post-agent)

Two issues surfaced during browser smoke-testing:

### Flag A — appearance cycler stopped cycling icons

**Cause:** Agent misread the 2252 spec ("Use this icon for the
dark/light/system/Happy mode control") as asking for a single static icon and
collapsed `getThemeIcon()` into a static `<AppearanceIcon>` wrapper. Figma
node `198:1789` (full Happy-mode sidebar) actually shows the Appearance row's
trailing icon as the Happyhour smiley — i.e. the trailing icon **is**
per-state (monitor/sun/moon/smiley).

**Fix** in `client/src/components/sidebar.tsx`:
- Restored the `Monitor, Smartphone, Sun, Moon` lucide imports + `HappyhourLogo`
  import; dropped `AppearanceIcon`.
- Restored `getThemeIcon()` with the original `system → Monitor/Smartphone`,
  `light → Sun`, `dark → Moon`, `happy → HappyhourLogo` mapping.
- Swapped the render site back from `<AppearanceIcon />` to `{getThemeIcon()}`.
- Deleted `client/src/components/icons/appearance-icon.tsx`.

### Flag B — dark-mode logo face rendered white

**Cause:** Original component used `fill="currentColor"` on the face (smile +
checkmark-eye) paths, which picked up `text-foreground` — near-black in light
mode, near-white in dark mode. Canonical `client/public/favicon.svg` uses
`fill="black"` explicitly; Figma node `114:1332` (dark-mode header) shows the
mark with **yellow circle + black face**.

**Fix** in `client/src/components/icons/happyhour-logo.tsx`:
- Face path `fill` changed from `currentColor` to `#000000` for both paths.
- This made the `"dark"` variant identical to `"default"` (both yellow circle
  + black face), so simplified the union from `"default" | "dark" | "happy"`
  to `"default" | "happy"`.
- `client/src/pages/world-clock.tsx` — `logoVariant` logic collapsed from a
  3-way ternary to `resolvedTheme === "happy" ? "happy" : "default"`.

## City loading: three tiers

**Tile cache** (`client/src/lib/tile-cache.ts`) — persists full
`TimezoneOption` metadata per selected zone under `localStorage["happyhour:tile-cache"]`.
New helper `getCityOrCachedTile(key)` falls through to the cache when neither
top nor full dataset is yet loaded. Cache writes on mount (after any tier
ready) and on `selectedZones` / `heroZone` change. Returning users now render
tiles synchronously on first paint even for cities outside the top-500.

**Top-cities bundle** — new `scripts/build-top-cities.mjs` derives a 38.97 KB
JSON with the top 500 cities by population + compact deref tables. Statically
imported by `loadTopCities()` in `city-lookup.ts` so it lands in the initial
JS payload. Search dropdown is responsive from keystroke 1.

**Lazy full dataset** — `loadCities()` is no longer called on mount. It
fires on `requestIdleCallback` (timeout 3 000 ms) OR on first open of the
Add Time Zone dropdown, whichever happens first. `loadCities()` is idempotent
(memoized promise).

New npm script `build:top-cities`. Re-run when `cities.json` is regenerated
from GeoNames.

## Error-state design audit (later session)

Quick inventory of failure surfaces in the app, ranked by user visibility.
None are currently designed — most fail silently or fall through to defaults.

| # | Error state | Where it triggers | Current behavior | Worth designing? |
|---|---|---|---|---|
| 1 | Weather fetch fails | Open-Meteo down / offline / rate-limited | `useWeather` throws; icon/temp missing | Yes — most visible, flaky wifi |
| 2 | City search: no results | User types a city that doesn't match | Empty dropdown | Yes — common, cheap win |
| 3 | Offline / no network | Global | Weather + city-DB lazy load fail silently | Yes — at minimum a subtle banner |
| 4 | Empty state (no cities) | First load | Local clock tile only | Yes — onboarding nudge |
| 5 | Geolocation denied | Closest-city detection | Silent timezone-derived fallback | Low — already graceful |
| 6 | Full city DB fails to load | Tier-3 lazy load | Falls back to cache/bundled subset | Low — degraded but functional |
| 7 | Cloud sync failure | `use-cloud-sync` | `syncStatus = "error"` — not surfaced to user | Medium — user should know |
| 8 | Invalid time input | Clock edit (known buggy per memory) | Duplicate AM/PM etc. | Yes — tied to open refactor |
| 9 | 404 / unknown route | Wrong URL | Generic dev-flavored `not-found.tsx` | Yes — polish copy |

**Top three to design first:** weather failure (happens most), search no-results
(cheapest win), empty state (first-impression moment). The rest are either
already graceful or lower-traffic.

needed in `world-clock.tsx` or the hook beyond the dead-`??`-removal above.

### Pending user action

The migration and API deploy are coupled — once the new API is deployed it
expects the column. Apply the migration first, then deploy.

```bash
# From api/ directory
npx wrangler d1 migrations apply happyhour-db --local   # dev
npx wrangler d1 migrations apply happyhour-db --remote  # prod
# Then deploy the worker
npx wrangler deploy
```

Existing rows get `show_rel_time = 0` from the column default — no data loss
or backfill needed.

### Soft-rollout note

If the API is deployed before the migration, every PUT will fail because the
INSERT references a column that doesn't exist. GETs would still work but
return rows missing the column, and the client would crash on the now-required
`showRelativeTime`. Order matters: migration → deploy.
