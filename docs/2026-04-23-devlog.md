### Mobile drag saga (commits `1d97c39`, `0da5bff`, and the final `<pending>`)

Long user-reported issue: tile-body long-press activates (yellow background) but finger-drag scrolls the page instead of moving the tile. Three iterations:

1. **`1d97c39`** — removed `touch-action: manipulation` from the tile wrapper, reasoning that it was overriding the body-level `touch-action: none` set in `handleDragStart`. `touch-action` doesn't cascade up from body; an element's explicit value sticks. Partial improvement only.
2. **`0da5bff`** — set `touch-action: none` on the tile wrapper **always**. Blocks iOS from claiming the touch for scroll at all. Eliminated the race, but the user still hit intermittent drag failures after a few successful tries, and lost the ability to scroll the page by touching a tile (costly on a 1-column mobile layout where tiles fill the viewport).
3. **Final iteration** — fundamental approach change: on touch devices, disable tile-body long-press entirely and route all mobile drag through the grip-handle icon. The `HandleTouchSensor` (150 ms, 8 px, `[data-drag-handle]`-scoped) was already the reliable path; the `TileTouchSensor` was the flaky one. Detection via `window.matchMedia("(pointer: coarse)")` mirrors the sidebar's existing `isTouchDevice` check.

Final code changes (the handle-only pattern):

- `client/src/components/time-zone-converter.tsx:257-278` — extracted `tileSensor` to a separate `useSensor(...)` call; `useSensors(...)` now conditionally includes it via `isTouchDevice ? null : tileSensor`. dnd-kit's `useSensors` filters out null entries.
- `client/src/components/digital-clock.tsx:427-438` — added `[@media(pointer:coarse)]:p-2.5` to the drag-handle container so the touch target grows to ~40 px on mobile while desktop keeps its tight `pt-2.5 pr-1` footprint.
- `client/src/index.css:244-259` — removed `touch-action: none` from `[data-testid^="draggable-zone-"]` (tile wrapper); added `[data-drag-handle] { touch-action: none; }` so the handle still wins its 150 ms race. Users can now scroll the page by touching tile bodies on mobile again.
- `docs/PRD.md` — updated the Drag-and-Drop Revisions scope: noted the desktop-only applicability of tile-body long-press, added the mobile-drag policy bullet, and closed bug #116 (touch drag unreliable on left-most column) which is resolved by this policy.

## Why the final iteration should stick

Browsers with long-press-drag-on-card patterns that don't fail on iOS (Trello, Notion, Todoist) all block native scroll on the card. The Happyhour layout's tiles fill the mobile viewport, so blocking scroll on tiles gets costly — hence the new approach: drag only from a narrow, deliberate target (the handle). The 150 ms activation means users who touch the handle are committed to dragging before iOS commits the touch to anything else, so the race stops existing. Desktop is unchanged — the mouse sensor covers the whole tile as before.

### Fix: cookie bar specificity + theme tokens (`16892d7`)

Two compounding bugs in the first-pass CSS:

- **Specificity.** Silktide's library CSS has a `@media (max-width: 640px)` rule using `#stcm-wrapper #stcm-banner` (specificity 0,2,0). Our initial overrides used bare `#stcm-banner` (0,1,0) and silently lost — so on mobile the bar rendered as Silktide's default 600px floating pill with 12px side margins instead of a full-width bottom bar. Every banner rule now prefixes with `#stcm-wrapper` to match Silktide's specificity; vite-injected styles still win cascade on ties.
- **Theme tokens.** The Happy-mode Figma inverts the button colors (dark `#333` bg with `#efefef` text) because yellow-on-yellow is unreadable. Initial CSS kept yellow buttons in all three modes. Dark-mode background + text tokens were also off vs. Figma variables.
