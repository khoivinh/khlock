## 1. Cookie banner + modal CSS

All in `client/src/styles/silktide-overrides.css`. Silktide JS untouched.

### Silktide-S visibility

The banner's `.stcm-logo` SVG embeds `<path fill="inherit">` and the parent `<a>` had `fill: var(--textColor)` set. Some browsers don't propagate `fill="inherit"` cleanly when the chain involves a CSS custom property at the ancestor — diagnosis: render appeared empty or transparent. Force the path color directly:

```css
#stcm-wrapper #stcm-banner .stcm-logo svg path {
  fill: var(--textColor);
}
```

Verified against Figma 280:4223 (light mode → dark `#333`), 280:4306 (dark → `#efefef`), 280:4461 (happy → dark `#333`).

### Buttons — vertically-centered text

Figma's `pt-8 pb-10` design intent left text 1px above the visual center at runtime (per user feedback). Switched to symmetric:
- Banner desktop: `padding: 9px 12px`
- Banner mobile (`<640px`): `padding: 7px 10px`
- Modal: `padding: 9px 12px`

### Focus rings

Default Silktide focus is `box-shadow: 0 0 0 2px var(--backgroundColor), 0 0 0 4px #ffffff` — a 4px white halo that's far too heavy on a dark modal close-X (visible "ring" was bigger than the 14×14 X itself). Override per spec to a thin 1px #333 outline on:
- `.stcm-button`, `.stcm-button-primary` in both banner and modal
- `.stcm-preferences-button` in banner
- `.stcm-modal-close` (with `outline-offset: 2px` so the outline doesn't crowd the X bars)

### Modal close-X spacing

Figma's "Close Affordance" container has `pl-[40px]` between the description block and the X. The previous impl relied only on header `gap: 9px` — too tight. Added `margin-left: 40px` to `.stcm-modal-close`, mirroring Figma. Total separation from description right-edge to X left-edge is now 9px (gap) + 40px (margin) ≈ 49px.

### Modal corner radius

Per spec, match the sidebar (`sidebar.tsx:245` uses `rounded-[15px]`). Bumped `#stcm-modal { border-radius: 10px → 15px }`.

## 2. Ancillary header parity

**Root cause:** `site-page-layout.tsx` hard-coded the logo + wordmark at 38×38 / 43px via inline `style`. The home page (`world-clock.tsx:58`) applies `mobileScale = 0.73` when `window.innerWidth < 500`, landing the lockup at 27.74×27.74 / 31.39px on mobile. Discrepancy created visible "movement" of the lockup when navigating between home and `/about` `/privacy` `/support`.

**Fix:** Drop the inline `style` and use Tailwind utilities:

```tsx
<HappyhourLogo
  className="shrink-0 size-[38px] max-[499px]:size-[27.74px] max-[499px]:mt-[2px]"
/>
<HappyhourWordmark
  className="shrink-0 h-[43px] max-[499px]:h-[31.39px] w-auto"
  style={{ color: wordmarkColor }}
/>
```

Ancillary pages don't have the home page's scroll-driven shrinking, so a CSS-only `max-[499px]:` breakpoint suffices — no JS scroll listener needed.

## 4. Mobile favicon

`favicon.svg` was the only icon shipped, with both `rel="icon"` and `rel="apple-touch-icon"` pointing at it. Mobile Safari doesn't reliably honor SVG for `apple-touch-icon`, and Android Chrome wants a manifest with PNG icons.

Used macOS `sips` to convert the SVG to three PNGs:

```bash
sips -s format png -z 180 180 client/public/favicon.svg --out client/public/apple-touch-icon.png
sips -s format png -z 192 192 client/public/favicon.svg --out client/public/icon-192.png
sips -s format png -z 512 512 client/public/favicon.svg --out client/public/icon-512.png
```

Added `client/public/manifest.json` listing the 192 + 512 PNGs (also sets `theme_color: #ffd900` and `start_url: "/"`).

Updated `index.html`:
- Kept `<link rel="icon" type="image/svg+xml" href="/favicon.svg">` for desktop browsers that prefer SVG
- Replaced `apple-touch-icon` with `<link rel="apple-touch-icon" sizes="180x180" href="/apple-touch-icon.png">`
- Added `<link rel="manifest" href="/manifest.json">`
